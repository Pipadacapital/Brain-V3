# Failure Mode & Effects Analysis (2026-06-19)

> Principal-level FMEA for the Brain AI-Native Commerce OS monorepo. Every row is anchored to a verified finding with a real `path:line`. Severity/Likelihood/Detectability are scored 1–10 (10 = worst); **RPN = S × L × D**. Sorted by RPN descending.
>
> Scoring convention:
> - **Severity** — production blast radius if the mode fires (10 = total tenant-wide data/revenue loss).
> - **Likelihood** — probability of occurrence over a 6-month operating window given current code/infra state (10 = near-certain / already true).
> - **Detectability** — how *hard* it is to detect (10 = silent, no alert/CI/test catches it; 1 = caught immediately by an existing gate).
>
> Findings that describe a *current static-state defect already 100% present* (e.g. inert ESLint rule, stub module) are scored Likelihood high because the latent condition exists today; the failure they enable is what is rated for Severity.

## Headline verdict

**No P0 runtime-active data-corruption mode was proven.** The dominant risk class is **silent enforcement gaps + missing deployment/scheduling infrastructure that block production operation and remove cross-tenant isolation backstops** — high *Detectability* scores (silent at CI, no alert) are what push RPN. The single highest-RPN mode is **collector accept-availability coupled to Postgres** (ARC-3): the spool is the durability anchor and any Postgres unavailability fails event acceptance for *all* brands simultaneously, with the documented ADR-003 failure model now incorrect.

## Cross-cutting observations

- **Detectability is the dominant RPN driver.** 11 of 19 modes score Detectability ≥ 7 — they are silent at CI and emit no runtime alert. The portfolio's biggest leverage is *adding gates*, not rewriting logic: a duplicate-prefix CI lint (#10), a barrel-resolution test (#7), a StarRocks-pool-factory test (#5), the boundaries TS resolver + descriptor reorder (#2, #5), and `finalized`-count / parity-silence alerts (#3).
- **Two independent root causes collapse the entire metric-engine fence** (#2 descriptor ordering, #5 missing `@brain/*` resolver). Either alone makes the fence inert; both are present. This is the single most important multi-tenant isolation backstop and it is currently a no-op guarding 9 live imports.
- **The CD path is non-functional end-to-end** (#3 no schedulers, #4 no Dockerfiles/Helm + skipped scans). Nothing can be shipped to prod via GitOps today; this is masked only because app-of-apps is not yet activated.
- **No proven P0 runtime data-corruption mode.** The brand predicate is present in every cited isolation-sensitive caller (credit-writer, repull GUCs); the risk is the *removal of the CI/test backstop* that would catch a future regression, not an active leak.

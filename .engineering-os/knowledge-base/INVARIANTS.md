# INVARIANTS — the non-negotiables (Brain, Phase 1)

> These are the **product-specific** invariants for Brain, layered ON TOP OF the OS Iron Laws
> (`engineering-os-blueprint/00-first-principles.md §2`). A change that would break any invariant
> below is a Foundation amendment — not a normal requirement — and requires a Stakeholder-logged
> waiver before any implementation begins. The Security Reviewer and Engineering Advisor hold a VETO
> on any PR that violates these invariants.
>
> Sources: BRD §2, §18, §20; doc 08 §3–§7, §13; doc 07 §4, §25, §26; doc 09 §B; doc 12 §1, §4, §11.
>
> This file is the merge of: (a) security/compliance invariants from the Engineering OS Foundation
> security review, (b) engineering-principle invariants from doc 12 §1, and (c) structural
> invariants from the Architecture docs.

## Invariants table

| # | Invariant (must always hold) | Enforced by |
|---|---|---|
| **I-S01** | **Brand isolation is absolute and structural.** A query, event, cache lookup, or AI context may only return data belonging to the `brand_id` of the authenticated request. A cross-brand data exposure is a P0 incident (SLO = 0, breach workflow, no acceptable occurrence rate). Isolation is enforced at every layer independently — not by any single mechanism. | Postgres RLS (`USING (brand_id = current_setting('app.current_brand_id')::uuid)`) + non-owner role (no `BYPASSRLS`) + middleware GUC assertion before every query; Iceberg per-brand S3 prefix + per-brand KMS DEK; StarRocks per-brand row policies + Analytics API as the sole reader; Redis `tenant-context.brandKey()` helper (raw keys lint-banned). CI isolation-fuzz at every layer (cross-brand query must return nothing, not another brand's data). |
| **I-S02** | **No raw PII in events, logs, marts, or caches — ever.** All operational tables, event payloads, log lines, StarRocks marts, and cache entries carry only `sha256(per-brand-salt ‖ normalized value)` hashed identifiers or vault references (`pii_vault_reference`). Plaintext contact PII lives only in `contact_pii` (KMS-encrypted vault, accessible to `send_service` role only). NLQ queries are stored in `ai_provenance.question_redacted` form only. | No-PII schema-lint CI gate (doc 07 §26 gate 6); logger middleware redaction at the logger and at the log-shipping layer; `contact_pii` RLS (`app.role='send_service'` required); nightly log-grep in staging (zero PII hits required — any hit pages Security). |
| **I-S03** | **Consent is enforced by construction — a non-consented contact is structurally impossible.** The compliance engine's `can_contact()` check is hard-coded into every outbound path (email, WhatsApp, push, CAPI passback). `opt-out` / `withdrawn` overrides all marketing channels. A feature flag or configuration may not bypass this check. Consent withdrawal suppression propagates within 15 minutes (SLO = 0 late deliveries post-withdrawal). | `consent_tombstone` + `consent-suppressor` consumer (fast-path, fail-closed, <15 min SLA); compliance engine on every send path; `pending_window` queue checks tombstone before flush; CI consent-suppression integration test (a post-withdrawal send attempt must be blocked). |
| **I-S04** | **Consent withdrawal is retroactive.** A withdrawal must suppress all pending outreach, trigger deletion of already-passed-back conversion data from ad platforms (CAPI deletion signal), and be logged with the consent state. The `consent_record` table is append-only; no row is deleted on withdrawal — a new row with `state='withdrawn'` is inserted. | `privacy.consent.withdrawn` event → `consent-suppressor` (suppresses pending sends) + `capi-deletion` consumer (ad-platform deletion); `consent_tombstone` table; CI integration test asserts retroactive suppression. |
| **I-S05** | **Erasure = crypto-shred.** A DPDP/PDPL erasure request results in: (1) destruction of key material in `brand_keyring` (KMS key deletion renders `contact_pii` ciphertext permanently unreadable); (2) tombstone of the `customer` node to an opaque `surrogate_brain_id`; (3) re-projection of marts onto the surrogate; (4) erasure-aware Iceberg compaction (old snapshots must not resurrect identifier hashes); (5) CAPI deletion signal. The ledger and audit log survive on the surrogate — the math reconciles; the person is forgotten. Loss of the `brand_keyring` DEK is a permanent, unrecoverable crypto-shred event (crown jewel — quarterly restore drill required). | `crypto-shred` consumer + erasure job sequence; `pii_erasure_log.vault_shredded=true` gate; erasure integration test (post-erasure PII query returns nothing; ledger row carries surrogate). |
| **I-S06** | **Audit log is append-only, hash-chained, and WORM-anchored.** The `audit_log` table is write-once at the PostgreSQL GRANT level (app role: INSERT+SELECT only, no UPDATE/DELETE). Each row carries `entry_hash = sha256(prev_hash ‖ canonical(row))`. Hourly checkpoint hash is written to S3 Object Lock (WORM). A tampered row breaks every subsequent hash. The chain is verified quarterly; any break is a P0 incident. | PostgreSQL GRANT constraints; hash-chain write in `packages/audit`; hourly S3 Object Lock checkpoint job; quarterly chain-walk verification job. |
| **I-S07** | **Money is never a float.** All monetary values are stored and transmitted as `*_minor BIGINT` (integer minor units) paired with `currency_code CHAR(3)` (INR, AED, SAR). Float arithmetic on money is prohibited everywhere in the codebase. The metric engine always returns integer minor units. | Money-lint CI gate (any `*_amount`, `*_value`, `*_fee`, `*_cost`, `*_revenue` column typed as `float`, `double`, or unpinned `numeric` fails the build); enforced in TypeScript money types in `packages/core`. |
| **I-S08** | **AI never invents numbers and never writes data.** All numeric values surfaced by the LLM layer must trace to deterministic metric engine output recorded in `ai_provenance.metric_binding`. Asserting a figure not present in `metric_binding` or `value_minor` is a P0 honesty violation. The LLM/AI layer has no write tool, no DB credentials, and no path to modify data, model weights, or eligibility rules. The MCP server is read-only (no write tools registered in any phase). | NLQ resolution eval gate (CI golden query set — every number in LLM output must appear in `metric_binding`); MCP tool registry audit (every tool must declare `read_only: true`; a write tool is a Security VETO); `ai` module design: no DB write credentials. |
| **I-S09** | **Secrets never in DB rows, logs, or the codebase.** OAuth tokens, connector credentials, and DEKs are stored by reference only (`connector_instance.secret_ref` → AWS Secrets Manager; `brand_keyring.wrapped_dek` = KMS-wrapped, never exported plaintext). No column named `*_token`, `*_secret`, `*_key` holds a plaintext string. | Semgrep SAST rule (plaintext-token-column detector); gitleaks + TruffleHog pre-commit + CI secret scan; AWS Secrets Manager + IRSA for runtime credential fetch; KMS for DEK wrapping — DEK never leaves KMS in plaintext. |
| **I-S10** | **PCI SAQ-A boundary is inviolable.** Brain stores gateway tokens only. PANs, CVVs, raw bank account numbers, and full UPI secrets never enter any Brain data store, log, event, or in-memory buffer. A change that would bring card data into Brain's perimeter is a Security VETO and requires a full PCI-DSS scoping review. | `pan-cvv-column-lint` CI gate (columns named `pan`, `cvv`, `card_number`, `full_account` fail the build); schema review on every migration PR; `payment_method` table holds `gateway_token text` + `provider text` only. |
| **I-E01** | **Contract-first.** No code may be written for a behaviour until its Zod contract exists in `packages/contracts` and the codegen artifact (OpenAPI / types / Avro / MCP schema) has been generated and committed. A PR that changes behaviour without a prior contract change is blocked. | `packages/contracts` Zod→codegen gate (buf-breaking + Pact stub in CI); CODEOWNERS on `packages/contracts` requires the consuming-domain owner's approval. |
| **I-E02** | **Data-first and replayability-first.** A feature is not done until its data reconciles and is replayable from append-only Bronze. History is never mutated — every economic effect, reversal, and correction is a new signed row. No destructive migration on `bronze.*`, any ledger table, `audit_log`, or `decision_log`. | Doc 12 §1 (replayability-first principle); replay test (rebuild Silver/Gold from Bronze → identical) runs nightly and before every release; migration gate: any `ALTER TABLE … DROP COLUMN` or destructive operation on event/ledger tables fails CI review. |
| **I-E03** | **Deterministic-first — compute in code, not in a model.** Every number Brain computes lives in the TypeScript metric engine (`packages/metric-engine`). dbt/SQL provides additive input marts only. No metric may be computed in a prompt, an LLM response, a dbt macro that mirrors the engine, or an ad-hoc query. | Doc 12 §1 (deterministic-first); parity oracle CI gate (TS engine output vs independent SQL recompute on the same snapshot — any delta is a build failure); the metric registry is the only definition. |
| **I-E04** | **Measurement-first — every number traces to `metric_version` + snapshot, and the parity oracle is sacred.** A number may not be surfaced (on a dashboard, in a recommendation, in an LLM response, or via the Analytics API) if it cannot be reproduced by the parity oracle. | Parity oracle CI gate (doc 12 §1; METRICS.md Rules); every `ai_provenance` row binds to `metric_binding` + `snapshot_id`; parity drift alarms page on-call. |
| **I-E05** | **Simplicity-first — a new source or capability is a folder, not a new service (the Single-Primitive Rule).** New deployables, databases, ledgers, or services are rejected by default; the burden of proof is on addition. Within the monolith, bounded-context boundaries are enforced via import-lint (`apps/` never imports `apps/`; cross-module reach-around past `index.ts` is banned). | Doc 12 §1 (simplicity-first); import-boundary lint in CI (`turbo --affected` + eslint import rules); new-deployable proposals require an ADR + Principal Architect review + CTO approval. |
| **I-ST01** | **One Analytics API read path.** The Analytics API is the sole component that reads from StarRocks or queries Iceberg-derived marts. No other service, module, dashboard, NLQ handler, MCP tool, or export job touches StarRocks or the Iceberg Bronze layer directly. | STACK.md ADR-002; import-lint rules prevent direct StarRocks client instantiation outside `packages/analytics-api-client`; architecture review gate on any new data-access path. |
| **I-ST02** | **Accept-before-validate durable spool — the 99.95% guarantee lives here.** The Collector acks an event to the producer only after persisting it to the durable spool. Validation, schema checks, and downstream produce may fail; the event is never lost once spooled. | STACK.md ADR-003; collector integration test (schema-registry-down scenario — event still reaches Bronze); `events_captured_count` metric counts quarantined events. |
| **I-ST03** | **One metric definition, many consumers.** No metric is defined, computed, or approximated in a dashboard, report, LLM prompt, MCP tool, or dbt macro independently. Every consumer reads from the registered `metric_definition` keyed by `(metric_id, version)`. A definition change requires a version bump + `metric_audit` entry + CI parity re-run. | METRICS.md Rules; metric-engine CODEOWNERS requires 2 approvals (one must be the Data CODEOWNER); CI gate: any string literal matching a known metric formula outside `packages/metric-engine` fails linting. |
| **I-ST04** | **Idempotency on `(brand_id, event_id)` for every mutating endpoint and consumer.** Every inbound event consumer, connector UPSERT, and HTTP-mutating endpoint must be safe to replay with the same payload. Duplicate delivery never creates a duplicate economic effect. | Doc 12 §4 (code review checklist — idempotent?); connector acceptance criteria require idempotent UPSERT; integration test: replay of the same event batch produces no new ledger rows. |
| **I-ST05** | **One outbound notification / send-consent chokepoint.** All outbound communications (email, push, in-product, WhatsApp, CAPI passback) pass through the single named consent + DND + DLT + quiet-hours module before any channel adapter. No path bypasses this chokepoint. | STACK.md ADR-012 (M10); compliance engine's `can_contact()` is hard-coded, not injected; CI: send-path integration test asserts the chokepoint fires on every send attempt. |

## Anti-patterns

These are specific patterns that are prohibited in the Brain codebase. A PR introducing any of these is a blocker.

**Data access**
- No `OFFSET` pagination on any growable list (ledger, events, recommendations, audit log). Use keyset/cursor pagination.
- No app-to-app HTTP imports (one `apps/` deployable does not import or call another directly — only through defined API contracts).
- No cross-module reach-around past `index.ts` (a module in `core/modules/A` does not import internal files from `core/modules/B` — only the public `index.ts` surface).
- No raw Redis keys built outside `tenant-context.brandKey()`. Raw key construction in application code is lint-banned.

**Money**
- No float-typed money column or float arithmetic on monetary values anywhere in the codebase.

**AI / model**
- No model-written SQL (text-to-SQL is prohibited; the MCP surface is read-only metric/context retrieval only).
- No model-invented numbers (any numeric claim in an LLM output that is not present in `ai_provenance.metric_binding` is a P0 honesty violation).

**Data integrity**
- No destructive migration (`DROP COLUMN`, `TRUNCATE`, `DELETE` rows) on `bronze.*` event tables, `realized_revenue_ledger`, `attribution_credit_ledger`, `invoice`, `audit_log`, or `decision_log`. Corrections are new rows; ledger corrections are signed-negative rows.
- No direct `UPDATE` or `DELETE` on any ledger or audit table from the application role or any automated job.

**Security**
- Never grant `BYPASSRLS` to the application role or any role used by a service at runtime.
- Never replicate a brand's data outside its designated region without a DPA sub-processor registry entry and a Security VETO sign-off.
- Never skip the isolation-fuzz CI gate for any PR touching a data-access path, even a "minor" refactor.
- Never register a write tool on the MCP server in any phase — the MCP surface is read-only by invariant.
- Never bypass the compliance engine on any outbound path — a direct send that skips `can_contact()` is a compliance violation regardless of urgency.
- Never store a credential, token, or DEK as a plaintext string in a DB column, environment variable (visible in CI), or source file.

# METRICS — the single-source metric registry

> The **one** place every metric/calculation is defined. If a value is computed in more than one runtime/language, the definitions must be **identical** and parity is checked against an **independent oracle** (Iron Law: deterministic truth). Models never invent these numbers — they classify/explain/synthesize only.
>
> Scope: **Phase 1 as-built, recommend-only, DETERMINISTIC decisioning.**
> Currencies: INR (primary) / AED / SAR. Money = integer minor units (`*_minor BIGINT`) + `currency_code CHAR(3)`; never floats.
> Sources: BRD §10.3–10.4, §21; PFS §6.5–6.6, §7.1–7.4; Data Model §5.4, §7, §12; Decision Engine Parts 1, 7, 11.

## Rules

- **One definition, many consumers.** No metric is redefined in a dashboard, a report, an LLM prompt, or a MCP tool. Every consumer reads from the single metric engine output or the registered `metric_definition` (Postgres `metric_definition` table, keyed by `(metric_id, version)`). Changing a definition requires a version bump + `metric_audit` entry + CI parity re-run.

- **Money = integer minor units + `currency_code`.** Every monetary column is `*_minor BIGINT + currency_code CHAR(3)`. No floats for money, ever (lint-enforced, Data Model §4). FX conversion uses the `fx_rate` row pinned to the ledger row's `economic_effective_at` date — a sale leg and a later clawback leg of the same order may convert at different rates (correct accounting). A missing FX rate fails closed (no computation proceeds).

- **Quality-gated.** A metric whose inputs fail the data-quality gate is marked *estimated/untrusted* until the gate passes — never presented as authoritative. The authoritative gating table (PFS §6.3): below the **70 line** (below a C grade / cost-confidence `Insufficient`): numbers render "Estimated," high-risk recommendations are blocked, the slice is excluded from model training, and the CM2 affordability cap does not apply to billing. `effective_confidence = min(cost_confidence, attribution_confidence)` and the display names which leg is the problem (PFS §7.3; Decision Engine Part 7).

- **Finalized-only for decisions and billing.** Provisional/settling rows are labeled and visible but **never** feed billing, high-stakes recommendations, or the attribution closed-sum asserted by the parity oracle. The parity oracle continuously asserts hot-vs-finalized convergence within a defined window and alarms on breach (PFS §6.5).

- **The LLM never produces a number.** The LLM narrates the engine's output — it receives the computed `expected_impact_minor`, `effective_confidence`, and `evidence` and describes them. It does not compute, calculate, re-derive, or invent any metric value. A response asserting a number not present in the engine output is a P0 honesty violation (Decision Engine Part 11; PFS §11.4.3).

- **Parity oracle is CI-blocking.** The parity check runs the `metric_test` golden fixtures against an independent SQL recomputation over the canonical stores. Any drift between the engine output and the independent oracle fails the build. This is a hard launch requirement (PFS §10.9; Data Model §5.4).

- **Attribution closed-sum is enforced.** `Σ channel_contribution_minor + unattributed_minor = realized_revenue_minor` for every brand-period pair. The unattributed residual is **always rendered** alongside attributed numbers — never hidden, never spread silently across channels (BRD §10.8; PFS §7.4–7.5).

- **Append-only ledger, dual-date.** `economic_effective_at` drives attribution + CM2 as-of math. `billing_posted_period` determines which open billing period a late adjustment posts to; closed/invoiced periods are **never edited** (Data Model §7.3). The sale row is never mutated; every economic effect is a new signed row.

- **Identity scope is brand-scoped and never global.** `brain_id` carries a per-brand salt; hashes cannot be correlated across brands. The `identity_match_rate` denominator excludes bot-flagged sessions. No metric crosses the brand boundary.

> ASSUMPTION: **Realized-revenue horizon defaults** — approximately 25 days for COD orders and 7 days for prepaid, until the brand's own Fingerprint matures (cold-start uses category-benchmark P80). Exact defaults are an open decision requiring Data Engineer confirmation. Reference: BRD §10.3, §23.3.

> ASSUMPTION → PARTLY RESOLVED (doc 12 §4/§5): **Two distinct tolerances.** (1) The **parity oracle** (TS engine vs independent SQL recompute on the *same* ledger snapshot) targets **exact integer equality** — not a percentage. (2) **Reconciliation** (Brain-attributed revenue vs the Shopify order ledger, net RTO/refund) carries a tolerance of **±2–3%** by W4, with **>±5% = stop-and-fix**; BRD §8.5 cites orders ~1% / spend ~2% at connector level. The exact reconciliation tolerance is **frozen in Sprint 0** — owner: Data Engineer. Only that Sprint-0 freeze remains open.

## Effort-tier defaults (cost routing)

Every metric and every number produced by Brain is **tier-0 deterministic** — computed by the TypeScript metric engine over certified stores, never by a model. The cost-routing ladder (cheapest-sufficient-effort doctrine; `cost-routing-paradigms` skill; Engineering OS Blueprint §11):

| Tier | Relative cost | What Brain does here |
|---|---|---|
| **0 — Deterministic logic** | ~1 | **All metric computation.** Aggregations, as-of sums, ratio math, confidence minima, parity oracle, closed-sum check, detector trigger evaluation, priority scoring, conflict resolution. This is the only tier that produces numbers. |
| **1 — Statistical / ML** | ~100 | RTO-likelihood model (probability score input to provisional-credit discounting); anomaly thresholds (σ-based baselines for detector triggers); future: cohort survival, LTV forecast. Numbers from these are inputs to deterministic formulas, not direct outputs shown to the user. |
| **2 — Small model** | ~1,000 | Bounded NL classification (e.g. ticket triage on a bounded label set). Not used for metric computation. |
| **3 — Frontier / large model** | ~10,000 | Narration only: the `title`/`summary` fields of a recommendation, Morning Brief prose, NLQ conversational framing. The model receives the engine output (computed numbers + confidence + evidence) and describes it — it does not derive it. |

**Cost-routing discipline:**
- A large-model call where deterministic/ML/small-model would do is the most expensive engineering mistake and is blocked at design review.
- Every model call routes through the LiteLLM gateway (cheapest-sufficient, eval-gated per `llm-evals` skill); model calls outside the gateway are blocked.
- Prompt caching is assessed on every model call; a stable system-prompt prefix (tool catalogue + metric schema context) without a cache marker is a cost bug.
- Per-tenant monthly model-spend caps are enforced as gateway virtual-key budgets; above cap, narration degrades gracefully to a template (metric numbers are always available because they are deterministic and do not consume the model budget).
- Large-model creep above ~1% of total calls triggers a high-priority incident: the AI/ML Engineer audits prompts and the gateway routes.

**The "agentic AI" discipline (Decision Engine Part 11; BRD §11.3):**

| What sounds like "AI" | What it actually is | Tier |
|---|---|---|
| "Brain detects an RTO spike" | RTO rate > baseline + kσ (deterministic threshold) | Tier 0 |
| "Brain scores expected CM2 impact" | `expected_impact_minor` from the registered detector rule | Tier 0 |
| "Brain ranks recommendations" | `PriorityScore = (I_norm × U × effective_confidence) / E` (formula) | Tier 0 |
| "Brain predicts RTO likelihood" | Probabilistic model over pincode/courier/payment features | Tier 1 (ML) |
| "Brain writes the Morning Brief" | LLM narrates the engine's ranked decisions in prose | Tier 3 (large model — justified) |

**Eval gate:** any prompt or model change must clear the golden-set eval suite (NLQ resolution accuracy, faithfulness/groundedness, no fabricated numbers) before merge. A model wired into a routing tier without a passing eval is a blocker (`llm-evals` skill).

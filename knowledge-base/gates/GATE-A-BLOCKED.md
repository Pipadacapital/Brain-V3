## BLOCKING FAILURES

### ❌ BLOCKER 1 — A.5.3 ambiguity: **zero conflict rows** (criterion fails outright)
**Criterion (A.5 #3 / §A.2.3.4):** shared-device / `email→X, anon→Y` sessions must produce a `silver_stitch_conflicts` row and **NO** `silver_session_identity` stitch.

**Evidence (Trino, live):**
```
SELECT count(*) FROM iceberg.brain_silver.silver_stitch_conflicts;  ->  0
```
`silver_stitch_conflicts` is **empty** across all three golden brands. The `shared_device_family` scenario is designed into the generator (`packages/testing-golden/src/scenarios.ts:153-160` — one shared anon, two member emails, no purchase) precisely to force the "never guess" conflict path. The pipeline emitted **0** conflict rows, so the required "conflict row present" half of A.5.3 is **not satisfied**. There is no evidence in the current output that shared-device ambiguity is being detected at all.

### ❌ BLOCKER 2 — WA-16 stitch job is **non-reproducible on golden** (MERGE_CARDINALITY_VIOLATION)
**Impact:** blocks A.5.1, A.5.3, A.5.5, A.5.8 reproducibility + violates §1.6 (derivations are deterministic; reruns yield identical outputs) and §0.2.1/§0.2.3.

`db/iceberg/spark/silver/silver_session_identity.py` (the critical-path Stitch v2 job, WA-16) **aborts** on a clean `FULL_REFRESH=1` over the golden dataset — reproduced twice (default config and with recency disabled):
```
db/iceberg/spark/silver/run-silver-session-identity.sh  (FULL_REFRESH=1)
-> flag gate: 3/5 brand(s) have stitch.v2 ON
-> ERROR Executor: [MERGE_CARDINALITY_VIOLATION] The ON search condition of the MERGE
   statement matched a single row from the target table with multiple rows of the source table.
-> {"job":"silver-session-identity","status":"fail","duration_ms":11127,...}
-> EXIT=1
```
(logs: `/tmp/stitch-v2-default.log`, `/tmp/stitch-v2-fullrefresh.log`.)

**Root cause — session_key (murmur3_32) hash collisions.** The job derives the MERGE key as
`session_id = concat_ws(':', brain_anon_id, session_key)` (`silver_session_identity.py:204`) and MERGEs on `(brand_id, session_id)` (`:347`, `:371`). But `session_key` is a **32-bit** murmur hash of `session_id_raw`; distinct raw sessions on the same visitor collide, so one `session_id` maps to **multiple** `session_id_raw` → multiple source rows with different `session_start`/`event_date` → Spark MERGE cardinality violation. Confirmed on live golden (`silver_touchpoint`):
```
(brain_anon_id, session_key) pairs with >1 distinct session_id_raw:
  a0a0 = 5,  b0b0 = 3,  c0c0 = 3   (11 total)
example: anon a651abc9… session_key 998372190 -> [68bcd79f…, d474fbbd…]
```
This is a genuine code defect, **not** an environmental artifact: 32-bit birthday collisions are expected at golden volume (~thousands of sessions, √(2³²)≈65k). The existing 9,099 `silver_session_identity` rows are a **stale residue** of an earlier partial run and cannot be regenerated by the job as written.

**Fix (recommended):** MERGE on the truly-unique session identity. Carry `session_id_raw` through `_session_map`/`_session_identifiers` and make the target key `(brand_id, brain_anon_id, session_id_raw)` (or `session_id = concat(brain_anon_id, ':', session_id_raw)`), keeping `session_key` as a non-key column. Add a de-dup/`dropDuplicates` guard on the MERGE source and a unit assertion that the source has no duplicate MERGE keys. Re-run `FULL_REFRESH=1`, then re-verify A.5.1/A.5.3/A.5.5.

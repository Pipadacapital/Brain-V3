# TRIGGER SURFACES — Brain (Phase 1 as-built)

> The OS supplies the **categories**; this file supplies Brain's **concrete surfaces + thresholds**.
> Any change touching a trigger surface routes to the **high-stakes lane** (architecture + security +
> QA + final review + 2 personas + mutation tests) and is re-scanned post-build (a change that grows
> into a trigger surface mid-build is upgraded automatically). Sources: doc 04 §6.6/§7.7/§8/§10/§11/
> §12 Part F, doc 06 §1/§8. Money = `*_minor BIGINT` + `currency_code CHAR(3)` ∈ {INR, AED, SAR};
> tenant key = `brand_id`. Paths reference `apps/core/src/modules/*` and `packages/*`.

| Category (OS-fixed) | Concrete surface in Brain | Threshold / notes |
|---|---|---|
| **Compliance / regulatory boundary** | Consent (`notification` chokepoint consent enforcement + at-capture consent snapshot on the event envelope); erasure / crypto-shred (the `identity` crypto-shred job + Argo retention/erasure jobs + `POST /privacy/erasure-requests` → `erasure.requested`); DSAR access/portability (`POST /exports {scope:dsar}`); residency (the **RegionAdapter** in `ai`/LiteLLM + the sub-processor registry; in-region-by-default; cross-border model-provider paths). Regime = **DPDP (India)** now; PDPL/GDPR seams reserved. The **`tax_regime` enum** (`GST_IN | VAT_AE_5 | VAT_SA_15 | …`) and the **`region` column** are live Phase-1 model fields (doc 08 §36 Delta 1). | **any change** — incl. any change to a consent category, the quiet-hours window, DND/NCPR/DLT enforcement, the crypto-shred steps, the no-raw-PII-in-events lint (C2), a sub-processor/region pin, **or any change to the `tax_regime` enum values or region-routing logic**. |
| **Multi-tenancy / isolation boundary** | The 4 store layers: **Postgres RLS** policies + the non-owner app role (`packages/db` migration #1) · **per-brand S3 prefix** · **per-brand KMS** data keys (`packages/config`/Secrets) · **StarRocks row policies**. The seams (where leaks happen): the `brandKey()` / `tenant-context` cache helper (`packages/tenant-context`), the `brand_id` envelope assertion in `apps/stream-worker` consumers, the AI gateway/cache tenant scoping, the Owner-rollup stitch, the `X-Brand-Id` non-null assertion (`frontend-api`/API edge), MCP `brand_id × scope` checks. | **any change** to any RLS policy, row policy, KMS-key wiring, S3-prefix logic, `brandKey()`, the envelope `brand_id` assertion, or the `X-Brand-Id` gate. P0 isolation tests (API/DB/StarRocks/MCP) **must pass before any launch**. |
| **Authentication / authorization** | IdP integration (managed IdP/Authentik OIDC/SAML, MFA); the JWT mint/validate path; the **revocation denylist** checked on every protected action; the 4 permission templates (Owner/Brand Admin/Manager/Analyst) in JWT claims + RLS + MCP scopes; the `frontend-api` httpOnly-cookie↔short-token exchange + CSRF; MCP-key issuance/revocation. | **any change** — auth is never in app code; a permission/role/scope change, a token-lifetime change, or a denylist-check change is high-stakes. |
| **Money / financial calculation** | **`realized_revenue_ledger`** (recognition/reversal, dual-date rule) · the **CM2 / CM waterfall** in the metric engine (`packages/metric-engine`, `order_margin_fact`, True CM2) · the **billing meter** (`gmv_meter_snapshot` write-once, tier%/cap/min-fee, true-ups, invoices) · the **`attribution_credit_ledger`** (two-pass + proportional clawback) · all `*_minor` math + FX (`packages/money`). The metric engine is **fenced** — the only place a number is computed; dbt/SQL does additive marts only (CI flags non-additive functions in dbt). | **any change** — incl. the realized-revenue **horizon default (~25d COD / 7d prepaid)** + **reconciliation tolerance** (TODO-with-owner = **Data Engineer**), any rounding/allocation/FX rule, any ledger event-type, the dual-date rule, or the closed-sum assertion. Billing reads finalized rows only; closed periods immutable. |
| **Shared-contract parity** | The **TS metric engine vs the parity oracle** (`test:parity`): CI golden-fixture test (engine vs independent reference, blocking on drift) + the hourly runtime convergence monitor (StarRocks serving vs Iceberg/Bronze recompute) + the decision-path-purity assertion. Also the metric registry definitions (one definition everywhere) and `packages/contracts` (Zod-source-of-truth → OpenAPI/types, snake_case wire format identical to events + DB). | **any change** — to a metric definition, the metric engine math, a golden fixture, the reference implementation, a registry entry, or a shared contract schema. A cross-brand or parity regression fails the build. |
| **System-of-record / audit-log writes** | The **hash-chained, WORM-anchored Postgres audit ledger** (`packages/audit`; `hash(prev_hash ‖ entry)`, checkpoints anchored to S3 Object Lock; app role has no UPDATE/DELETE; a chain break is itself an alert). The **Decision Log** (recommendations + responses + **AI provenance**, immutable). `merge.committed`/`unmerge.committed` write audit + Decision Log. | **any change** — to the audit writer, the hash-chain mechanism, the WORM anchoring, the Decision Log writer, the AI-provenance record, or the set of audited actions (logins, role/permission/team changes, connector connect/disconnect, AI & MCP queries, exports, deletions, refunds, merges/unmerges, billing seals). The audit *event* is fan-out only — never the integrity source. |
| **Outbound side-effects** | The **single send/consent chokepoint** (`notification` module, `apps/core/src/modules/notification` — `notification`, `notification_pref`, `send_log`, `suppression_list`): email (primary), PWA push, in-product **now**; WhatsApp BSP + CAPI passback **Phase 3** (chokepoint built now). ALL outbound passes consent + `ai_processing` + withdrawal tombstone + DND/NCPR + DLT templates + the 9am–9pm window **fail-closed before any channel adapter**. Also: connector OAuth/sync writes to external systems (idempotent); exports; the Collector ingest edge. | **any change** — adding a channel, an adapter, a template type, a consent gate, the quiet-hours window, or the CAPI passback path. "Outreach to a non-consented customer is impossible" holds only with exactly one door — a second egress path is a hard violation (lint-enforced, ADR-012). |
| **LLM / model calls (product-specific)** | Any new **prompt / tier / routing** on a LiteLLM path (`packages/ai-gateway-client`, `apps/core/src/modules/ai`): a new prompt template, a model pin/swap/failover, an effort-tier change (deterministic ≫ statistical ≫ small ≫ frontier), a new NLQ binding, a new MCP tool, or a change to the untrusted-text envelope. | **any change** → the **eval gate** is a ship gate: the **NLQ resolution eval** (false-bind-rate → 0) + the **prompt-injection golden-set** must pass; a failover model serves only after passing its tier's resolution + narration evals. No text-to-SQL ever; numbers bind to `metric_id`; eligibility/confidence/metric fields are structurally unwritable by the model. |
| **Schema changes (product-specific)** | A migration touching **Postgres** (`packages/db`, `node-pg-migrate` — esp. RLS/app-role/ledger/audit/registry tables) · an **Iceberg** Bronze schema (Apicurio FULL_TRANSITIVE; additive-optional only; a breaking change → a new `.v{n+1}`) · a **StarRocks** Silver/Gold table or the reserved MMM-seam columns / `Calibrated` enum · an **event schema** (`packages/events`, Apicurio; the no-PII schema-lint). | **any change** — schema changes are reversible-migration-gated; FULL_TRANSITIVE compatibility on all Bronze-materialized streams; old events stay replayable forever; the reserved Phase-3 columns/enum must not be repurposed. |
| **Identity merge (product-specific)** | The `identity` module merge/unmerge/split path (`apps/core/src/modules/identity`, `packages/identity-core`): the `brain_id_alias` write, the union-find cycle guard, the versioned merge rule, the phone guard / shared-utility suppression, the async resolution writer, the centralized hashing shared library + its CI conformance vectors, the per-brand salt handling (the hardened trust zone). | **any change** — a merge/unmerge re-weights attribution (`merge.committed` → attribution re-credit + Customer-360 re-projection) and writes audit + Decision Log; the cross-brand-uncorrelatable negative test and the no-PII-in-logs test on this service must pass; false-merge-worse-than-missed is the safe default. |
